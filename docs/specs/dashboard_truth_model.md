# Dashboard Truth Model and Provenance Taxonomy

> Version: 1.0
> Created: 2026-03-08
> Purpose: Define how every data surface in the operator dashboard classifies its origin, freshness, and validation state.

## 1. Provenance Classes

Every piece of data displayed in the dashboard must be classifiable as one of:

| Class | Meaning | Visual Treatment |
|-------|---------|-----------------|
| `mock` | Generated by mock subsystem (`MOCK_MODE=true`), not from real runtime | Yellow badge |
| `real_local` | From real local runtime (process, file, database), but not from external platform | Green badge |
| `real_live` | From real external platform or service (RTMP target, TikTok API, real LLM call) | Bright green badge |
| `derived` | Computed from other data (e.g. uptime from start time, aggregated metrics) | Gray badge |
| `unknown` | Origin cannot be determined or has not been classified yet | Red badge |

### Rules

- No panel may display data without a classifiable provenance.
- `unknown` provenance must be visually distinct and never look like a healthy state.
- `mock` provenance must never look production-ready.
- Provenance is determined by the backend, not inferred by the frontend.

## 2. Validation States

The system-wide validation state is one of:

| State | Meaning |
|-------|---------|
| `unvalidated` | No validation has been run in this session |
| `validating` | Validation is currently in progress |
| `passed` | All required checks passed |
| `partial` | Some checks passed, some failed or were skipped |
| `failed` | Critical checks failed |
| `blocked` | Cannot validate because prerequisites are missing |

### Rules

- Validation state is tracked server-side, not UI-local.
- A `passed` state requires fresh evidence, not cached results from a previous session.
- `blocked` must list explicit blockers.

## 3. Runtime Modes

Each subsystem reports its runtime mode independently:

| Field | Values | Source |
|-------|--------|--------|
| `mock_mode` | `true` / `false` | `MOCK_MODE` env var via `is_mock_mode()` |
| `face_runtime_mode` | `mock` / `livetalking_local` / `livetalking_live` | LiveTalking manager state |
| `voice_runtime_mode` | `mock` / `fish_speech_local` / `edge_tts_fallback` / `voice_error` / `unknown` | Voice router active engine |
| `stream_runtime_mode` | `mock` / `idle` / `live` | RTMP streamer state |

### Rules

- If `mock_mode=true`, all subsystem modes should reflect `mock`.
- If `mock_mode=false` but a subsystem falls back to mock behavior, it must report honestly.
- `voice_runtime_mode=unknown` is the correct cold-start state before the first real non-mock synthesis resolves the engine.

## 4. Operator Action Receipt Schema

Every operator action (start engine, stop stream, switch product, etc.) must produce a receipt:

```json
{
  "action": "engine.start",
  "status": "success" | "error",
  "message": "LiveTalking engine started (pid=12345)",
  "timestamp": "2026-03-08T10:30:00Z",
  "provenance": "real_local"
}
```

### Rules

- No action may silently succeed or fail.
- Error receipts must include the error message, not just "failed".
- Receipts are ephemeral (displayed until next action), not persisted in this phase.

## 5. Panel-Level Truth Responsibilities

| Panel | Must Show |
|-------|-----------|
| Truth Bar (global) | `mock_mode`, `validation_state`, `last_validated_at` |
| Overview | system uptime with provenance, stream status with provenance |
| Readiness | each check with pass/fail + provenance |
| Engine | requested vs resolved model/avatar, engine state, provenance |
| Stream | RTMP target config status, stream runtime mode |
| Commerce | product source provenance (seed data vs real) |
| Monitor | metric source provenance |
| Diagnostics | health check provenance |

## 6. What Counts as REAL VERIFIED

A component is `REAL VERIFIED` only if:

1. It was tested with `MOCK_MODE=false`
2. It used real assets (not placeholders)
3. It produced observable output (not just "no error")
4. Evidence was recorded (command output, timestamp, result)

Everything else is at best `LOCAL VERIFIED` (works in mock/local) or `PARTIAL`.

## 7. Truth API Contract

The backend exposes a single consolidated truth endpoint:

**`GET /api/runtime/truth`**

Response shape:

```json
{
  "mock_mode": true,
  "face_runtime_mode": "mock",
  "face_engine": {
    "requested_model": "musetalk",
    "resolved_model": "musetalk",
    "requested_avatar_id": "musetalk_avatar1",
    "resolved_avatar_id": "musetalk_avatar1",
    "engine_state": "running",
    "fallback_active": false
  },
  "voice_runtime_mode": "mock",
  "voice_engine": {
    "requested_engine": "fish_speech",
    "resolved_engine": "unknown",
    "fallback_active": false,
    "server_reachable": false,
    "reference_ready": false,
    "last_latency_ms": null,
    "last_error": null
  },
  "stream_runtime_mode": "idle",
  "validation_state": "unvalidated",
  "last_validated_at": null,
  "provenance": {
    "system_status": "mock",
    "engine_status": "mock",
    "stream_status": "mock",
    "products": "derived",
    "metrics": "derived"
  },
  "timestamp": "2026-03-08T10:30:00Z"
}
```
